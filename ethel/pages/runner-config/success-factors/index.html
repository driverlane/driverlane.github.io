Core for Success Factors runners

These runners are related to working with Core for Success Factors.

The runners are:

  • CoreSFLoader
  • DocPipelinesSSFLoader

CoreSFLoader

Production

This runner will load objects into Core for Success Factors. It has the following configuration options:

  • addHash boolean - whether to add the currentHash lookup, defaults to true
  • checkDocType boolean - whether to check if the document type exists before uploading the document, defaults to false
  • connection mandatory object - the ID of the container where objects are loaded
    • authHost - server that provides authentication
    • contentHost - server that provides content functionality
    • subscriptionName - subscription name for the Core subscription
    • tenantId - tenant ID for the subscription
    • userId - user id from the API credentials for the Core subscription
    • userSecret - user secret from the API credentials for the Core subscription
  • logOnly boolean - whether to only log the actions without creating the files, defaults to false
  • timings boolean - whether to log the http timings or not, defaults to false
  • workspaceSearchField string - the field used in the employee workspace search, defaults to sf_person_information:user_id

It expects the following attributes in the message object, these can be remapped in the config attributes:

  • sourceFile string - the message attribute containing the path to the file to upload, defaults to sourceFile
  • docDate string - the message attribute containing creation date for the uploaded document, defaults to docDate
  • docType string - the message attribute containingthe document type for the uploaded document, defaults to docType
  • employeeId string - the message attribute containing the value used to find the workspace id, defaults to employeeId

It populates the following attributes in the new tasks it creates:

  • message.coreSFLoaderComplete the date and time the task is completed

DocPipelinesSSFLoader

Deprecated

The Document Pipelines integration to Core for Success Factors was full of errors when this was first created. The problems were eventually resolved, but in the time we were waiting we wrote the API based runner and would recommend using that instead. Doc Pipelines also has problems around multi-thread logging that make it a challenge when it comes to reporting so should be avoided unless there is no other option.

This runner isn't in the standard ethel repo, it's kept in the lifeblood-migration repo on Github.

This runner will create upload directories and files for the Document Pipelines Success Factors integration. It has the following configuration options:

  • addHash boolean - whether to add the currentHash lookup, defaults to false
  • copyFile boolean - whether to copy or move the file to the target location, defaults to true
  • loadLocation mandatory string - the path to the directory where the uploads are staged
  • logOnly boolean - whether to only log the actions without creating the files, defaults to false

It expects the following attributes in the message object, these can be remapped in the config attributes:

  • attributesFile string - the message attribute containing the path to the attributes file, defaults to attributesFile
  • commands string array - the message attribute containing the commands to be written to file, defaults to commands
  • sourceFile string - the message attribute containing the path to the file to upload, defaults to sourceFile

It populates the following attributes in the new tasks it creates:

  • message.docPipelinesSSFLoaderComplete the date and time the task is completed
