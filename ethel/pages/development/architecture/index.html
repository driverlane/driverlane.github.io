Architecture

The high level thinking about the ethel architecture is based on these basic customer environment assumptions:

  1. most engagements will involve one or many resources in a LAN as the source or target. This LAN may include third party cloud providers, but the resources will be firewalled off from the internet, often very aggressively.
  2. in many cases, ethel will have to run on a server with no access to the internet
  3. a few customers will require a docker based solution, running on their own servers
  4. but most customers will not care or constrain the solution and only point one and maybe point two is relevant for these customers
With these basic asumptions, we can see value in these flavours of ethel, created roughly in this order:
  • standalone Node.js solution, to support assumption two. Can also be used for assumption one until there's a cloud based ethel server.
  • Web client - Angular based on our experience with the framework
  • .NET agent able to talk to internet based ethel deployments, to support assumption one with a cloud based ethel server
  • dockerised verson of the standalone Node.js version to support assumption three
  • commodity cloud based queue and database solutions, to support assumption one, which will probably just be the dockerised version

We've worked through various forms of the server and you'll see some evidence of it in the repository, but at the moment we're focusing on the standalone Node.js version until we're happy with that. Docker versions of the core and client are available, but haven't been used in anger yet. We're expecting that the agent will be .NET solution to be more familiar to most organisations.

The Node.js standalone version (i.e. core in the repository) is a Nest.js solution. By default it uses the better-queue library to provide queue functionality for the task management and SQLite for the data storage.
