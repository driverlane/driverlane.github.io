Running ethel

A minimum configuration of ethel consists of a single instance of core. An instance of the client is the simplest tool for initiating and monitoring runs, but the API can also be used.

Command line

The simplest method to run ethel via the command line is to run the `build.ps1` script to generate the distributable versions of the core and client projects. You can then transfer these zip files to the server running the migration and extract them. The client folder includes configuration files for IIS and nginx as the web servers. This approach also makes it simple to create multiple instances of core as the core folder can be copied to multiple paths. Each core instance also needs a .env file with the encryption key and with multiple instances, the port number.

Docker compose

Instead of a simple command line initiation a number of example docker compose files are available in the harness folder and can be used as a starting point for a project specific docker file.

Configuration

If any client configuration options are created, they will available inside the client UI.

This page has configuration options for core - core configuration.

Queues and databases

By default ethel core uses better-queue as the queueing solution and SQLite as the database.

RabbitMQ can be used for queuing instead. RabbitMQ is required if you are looking to run multiple instances of core talking to a single database.

PostgreSQL and SQL Server can also be used for the database. SQLite is more performant than these, but should not be used when multiple instances of core are using the one database. File sizes of any of the database files should also be less than 350 gb. This is mainly due to file system constraints rather than SQLite constraints. Whichever databse platform is used, RAM and disk speed are important. Ideally you have enough RAM avilable to store the indexes in memory and SSD drives.

ethel agent

The ethel agent is intended to be a Windows service using a .NET application to connect to a RabbitMQ server and action tasks assigned to the queues it is consuming. It's only in initial development at the moment.
